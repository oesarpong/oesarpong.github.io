Introduction

The purpose of this lab was to gain exeriencing implement a PID controller. The controller will determine the response of the robot as it drives toward a wall, ideally stoping at a target/setpoint.

Prelab

To begin, I decided to setup my workflow and test to accomdate a reiterative procedure for tweaking gain controllers as well as debugging. To do this, I needed to establish a strong bluetooth connection in order to send the system inputs and recieving the sensor data to interpret it's response. Below is a summary of the final code used to accomplish this.

Data Processing

To start I created data arrays to store about 5 seconds worth of data and created a command GET_PID_DATA. I am mainly interested in how the "U" input and the position of the car would change over time. I made this a seperate callable command in order to decouple the time needed to send over data from the iteration time of the controller.

The data is recieved by my laptop via control_handler().

Additionally, I decided not to add this function into the main loop, but have it run only after being called which would reduce the looping time of my controller. This function is set to run for about 5 seconds, or until the robot remains in the same position for a set amount of time.

Positional Control

This section will cover the process that I used toward my final controller gains/inputs. The functions and algoirthims used during this process can be referenced in the Prelab, unless otherwise embedded. The first set of controls, are defined with a speed clamped to 25% of the max. This was done just to prevent any excessive collisions of the robot into the wall (see scale_u()).

High Level Controller Goals
  • Setpoint: 304mm/1ft
  • Effective within Starting Range: 2-4m
  • Minimize Settling Time & Overshoot
  • Small Overshoot is Allowable
  • < 1% Steady State Error

P Controller

I started my design with a P Comtroller.The proportional gain, KP, affects the speed of the controller in reference to the error. It reflects the ratio between the output and the input. Therefore, I started my controller off with a KP = 0.085. This value is the ratio of the maximum speed input over the median starting position (255/3000m). Ideally, I wanted to ensure that my robot would be running at maximum speed from the start of the command loop.

For my baseline controller, I decided to clamp my speed (using scale_u() ) at 25% of the max, which would help prevent any incidents or my robot smashing into the wall. Additionally, I increased my setpoint to 504mm to allow for a potentially large overshoot (set back in the next section). Below is resulting systems response:

PI Controller

Using the results from the P controller, I added an integrator, Ki, and implement a PI controller. Ki, sums the error over time, eliminating the steady state error if tuned correctly. One of the issues with adding an integrator term is that you run the risk of "Wind Up" which refers to an over accumlation of error beyond the systems saturation point, resulting in an input greater then the actuators capabilities. To avoid this, I added Wind-Up Protection by clamping the integrator term relative to the maximum speed input. See below:

Discussion

To my surprise, the PI controller didn't work completely as expected. I tried adjusting the gains and ended up settling at a Ki = 0.005. Interestingly enough the overshoot decreased and the steady state error increased. The integrator appeared to have a large contribution pushing the system farther away from the setpoint. Ultimately, the results did not go to waste. I decided that I would tackle the overshoot first and come back to the steady state error later.

PID Controller

Using the same PI controller gains, I added a derative gain, KD, to increase the sensitivity of my controller to change. KD works best with small "dt", and greatly reduces the overshoot of the system by responding sensitvely to change. Conversely, this also can bring some noise into the system which comes from a varying "dt" between loops. I attempted to filter as much noise as possible using denoise_d().

Curious about the integrator term, I also added a data array and sent it over bluetooth to see how it performed. Below are the results:

Discussion

The PID Controller performed the best accomdating for both speed, overshoot, and steady state error. After reaching a result I was happy with I decided to use this baseline and incrementally increase the speed limit to 100% of the max.

Final Controller

After configuring an acceptable controller, I incrementally increased the max speed of my system and tuned my gains to accomdate these changes. I did this by changing the clamped speed limit, and adjusted Ki/Kd gains to get a similar response as the previous section. Conversely, the proportional gain had an internal max speed as well, so once this was reached I increased the porportional gain to meet my user defined limit. Once met I could further tune the Kp to maintain max speed for longer. To verify my final gains, I conducted three trials for the same gains in order to compare the results.

Discussion

In conclusion, the gains I settled with Kp = 0.15, Ki = 0.0032, Kd = 0.071. With these results, it take my robot about 1.5 seconds to travel 2500mm which results in a Speed of 1.67 m/s.

I personally was satisifed with the results and was glad to see that they were repeatable. This controller prioitizes getting to the setpoint as fast as possible and avoids overshooting the setpoint.

Contribution of Surface Friction

For the final controller, I decided to change the surface that my robot traveled on. My solution was successful across multiple terrains, with slight variation in response. One of the variations that I found notable was the amount of slipping my robot experienced. On carpet there was a useful amount of friction that allowed the robot to stop in place, but on a smooth surface the robot slips and turns. This, in turn, effects the settling time and travel response.

Extrapolation Discussion

One of the largest limiting factors in this controller is the ToF's ranging time. Moreover, the effectiveness of the controller is dependent on the sampling rate in order to update the inputs as fast as possible. In order to speed up the sampling rate of the controller, it is encouraged to decouple its frequency from the sensor. Fortunately, this is something that I did in a previous lab by the using the last known data point of the ToF sensor wasn't ready when recording all sensor data. Alternatively, now to speed up the controller I isolated the measured data to only the front ToF sensor and used the linear extrapolation to predict new data.
